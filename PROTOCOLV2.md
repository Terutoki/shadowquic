# ShadowQUIC V2 协议规范

## 概述

ShadowQUIC V2 是一个高效的代理协议，基于QUIC传输层构建，支持TCP代理和UDP穿透。本协议是对V1版本的完全重构，移除兼容性负担，实现更高效的编码和更灵活的结构设计。

## 协议特性

- **标准QUIC VarInt编码**：兼容IETF QUIC标准，支持1/2/4/8字节变长编码
- **统一帧格式**：所有消息使用相同的帧头结构，最小仅15字节
- **0-RTT握手**：认证信息捎带在首个请求中，减少延迟
- **内置多路复用**：32位流ID支持42亿并发连接
- **原生扩展支持**：TLV格式扩展字段，无限扩展能力
- **统一UDP传输**：使用QUIC流承载UDP数据，简化实现

## 基础数据类型

### VarInt 变长整数

标准QUIC变长整数编码，最大支持2^62-1数值。

| 数值范围 | 编码长度 | 格式 |
|----------|----------|------|
| 0 - 63 | 1字节 | `0xxxxxxx` |
| 64 - 16383 | 2字节 | `10xxxxxx xxxxxxxx` |
| 16384 - 1073741823 | 4字节 | `11xxxxxx xxxxxxxx xxxxxxxx xxxxxxxx` |
| 1073741824 - 2^62-1 | 8字节 | `11xxxxxx xxxxxxxx xxxxxxxx xxxxxxxx xxxxxxxx xxxxxxxx xxxxxxxx xxxxxxxx` |

### SocksAddr 地址格式

兼容SOCKS5地址格式，用于表示目标地址。

| 类型值 | 地址类型 | 格式 |
|--------|----------|------|
| 0x01 | IPv4 | 1字节类型 + 4字节IP + 2字节端口 |
| 0x04 | IPv6 | 1字节类型 + 16字节IP + 2字节端口 |
| 0x03 | 域名 | 1字节类型 + 1字节长度 + 域名 + 2字节端口 |

## 帧结构

### 帧头格式

```
+--------+--------+--------+--------+--------+--------+--------+--------+---
|  Version (2)  |      Frame Type (62)      | Flags |   Stream ID    |
+--------+--------+--------+--------+--------+--------+--------+--------+---
|                    Length (VarInt)                                    ...
+------------------------------------------------------------------------+
```

| 字段 | 长度 | 说明 |
|-------|------|------|
| Version | 2位 | 协议版本，当前为1 |
| Frame Type | 62位 | 消息类型 |
| Flags | 8位 | 标志位 |
| Stream ID | 32位 | 流标识符 |
| Length | VarInt | 负载长度 |

### 帧类型定义

| 类型值 | 名称 | 描述 | 负载 |
|--------|------|------|------|
| 0x00 | ClientHello | 客户端握手 | ClientHello |
| 0x01 | ServerHello | 服务端握手 | ServerHello |
| 0x02 | Ping | 心跳请求 | u64时间戳 |
| 0x03 | Pong | 心跳响应 | u64时间戳 |
| 0x04 | Error | 错误通知 | ErrorFrame |
| 0x10 | Connect | TCP连接请求 | ConnectReq |
| 0x11 | ConnectAck | TCP连接响应 | ConnectAck |
| 0x12 | Data | 数据传输 | Bytes |
| 0x13 | Fin | 流结束 | 无 |
| 0x14 | Reset | 流重置 | u32错误码 |
| 0x20 | UdpAssociate | UDP关联请求 | UdpAssociateReq |
| 0x21 | UdpData | UDP数据 | UdpData |

### 标志位

| 位 | 名称 | 说明 |
|----|------|------|
| 0 | NEED_ACK | 需要确认 |
| 1 | IS_ACK | 是确认帧 |
| 2 | HAS_EXTENSIONS | 包含扩展字段 |
| 3 | FIN | 流结束标记 |

## 消息格式

### ClientHello 客户端握手

```
+--------+--------+--------+--------+--------+--------+--------+--------+---
| Version (8) |           Supported Features (64)                    |
+-------------------- ... 64 bytes ... ------------------------------+
|                         Auth Token (512 bits)                      |
+-------------------- ... 64 bytes ... ------------------------------+
|                     Extensions (TLV Format)                        |
+------------------------------------------------------------------------+
```

| 字段 | 类型 | 说明 |
|-------|------|------|
| Version | u8 | 协议版本 |
| Supported Features | u64 | 支持的特性位掩码 |
| Auth Token | [u8; 64] | 64字节认证令牌 |
| Extensions | Vec<Extension> | 扩展字段 |

### ServerHello 服务端握手

```
+--------+--------+--------+--------+--------+--------+--------+--------+---
| Version (8) |           Selected Features (64)                   |
+-------------------- ... 64 bits ... ------------------------------+
|                       Connection ID (64)                          |
+------------------------------------------------------------------------+
|                     Extensions (TLV Format)                        |
+------------------------------------------------------------------------+
```

| 字段 | 类型 | 说明 |
|-------|------|------|
| Version | u8 | 协议版本 |
| Selected Features | u64 | 协商后的特性 |
| Connection ID | u64 | 连接标识符 |
| Extensions | Vec<Extension> | 扩展字段 |

### ConnectReq 连接请求

```
+------------------------------------------------------------------------+
|                        Destination Address (SocksAddr)                |
+------------------------------------------------------------------------+
|                     Extensions (TLV Format)                          |
+------------------------------------------------------------------------+
```

### ConnectAck 连接响应

```
+--------+--------+
|   Status (16)   |
+-----------------+
|   Bind Address (SocksAddr)                                          |
+-------------------- ... ---------------------------------------------+
|                     Extensions (TLV Format)                          |
+------------------------------------------------------------------------+
```

| 字段 | 类型 | 说明 |
|-------|------|------|
| Status | u16 | 状态码 |
| Bind Address | SocksAddr | 绑定地址 |
| Extensions | Vec<Extension> | 扩展字段 |

### UdpAssociateReq UDP关联请求

```
+--------+--------+
| Has DST (8)   |
+-----------------+
|   DST Address (SocksAddr, optional)                                 |
+-------------------- ... ---------------------------------------------+
|                     Extensions (TLV Format)                          |
+------------------------------------------------------------------------+
```

| 字段 | 类型 | 说明 |
|-------|------|------|
| Has DST | u8 | 是否包含目标地址 |
| DST | SocksAddr | 固定目标地址，None表示任意地址 |
| Extensions | Vec<Extension> | 扩展字段 |

### UdpData UDP数据

```
+--------+--------+
| Has DST (8)   |
+-----------------+
|   DST Address (SocksAddr, optional)                                 |
+-------------------- ... ---------------------------------------------+
|   Payload Length (VarInt)                                            |
+-------------------- ... ---------------------------------------------+
|   Payload (Bytes)                                                    |
+-------------------- ... ---------------------------------------------+
```

| 字段 | 类型 | 说明 |
|-------|------|------|
| Has DST | u8 | 是否包含目标地址 |
| DST | SocksAddr | 目标地址，仅首次发送时需要 |
| Payload Length | VarInt | 数据负载长度 |
| Payload | Bytes | UDP数据负载 |

### ErrorFrame 错误帧

```
+--------+--------+--------+--------+
|         Error Code (32)       |
+------------------------------+
|   Message Length (VarInt)    |
+------------------------------+
|   Message (Bytes)            |
+------------------------------+
```

| 字段 | 类型 | 说明 |
|-------|------|------|
| Error Code | u32 | 错误码 |
| Message Length | VarInt | 错误信息长度 |
| Message | Bytes | 错误信息 |

## 通信流程

### 1. 握手阶段

```
Client                                          Server
  |                                               |
  |------------ ClientHello -------------------->|
  |                                               |
  |<----------- ServerHello ---------------------|
  |                                               |
```

### 2. TCP代理流程

```
Client                                          Server
  |                                               |
  |------------ Connect ----------------------->|
  |                                               |
  |<---------- ConnectAck ----------------------|
  |                                               |
  |============= Data ===========================|
  |                                               |
  |============= Fin ===========================>|
  |                                               |
```

### 3. UDP代理流程

```
Client                                          Server
  |                                               |
  |------------ UdpAssociate ------------------->|
  |                                               |
  |<---------- ConnectAck ----------------------|
  |                                               |
  |============= UdpData =======================|
  |                                               |
```

## 实现要求

### 编解码要求

- 所有消息必须实现 `SEncodeSync` 和 `SDecodeSync` trait
- 推荐实现 `SEncode` 和 `SDecode` trait 支持异步操作
- VarInt编码必须遵循标准QUIC规范
- 帧头必须先于负载编码

### 安全性要求

- 认证令牌使用SHA256(username:password)生成
- 推荐使用TLS加密传输
- 敏感信息不得记录日志

### 性能要求

- 最小帧头编码应在1微秒内完成
- 单连接吞吐量应达到10Gbps+
- 支持百万级并发连接

## 版本兼容性

- 相同主版本号必须互相兼容
- 服务端必须支持至少前一个次版本的客户端
- 版本不匹配时返回明确的错误响应

## V1 vs V2 协议对比

### 协议架构对比

| 特性 | V1 协议 | V2 协议 |
|------|----------|----------|
| **帧头结构** | 多部分组成：VarInt长度 + 请求类型 + 内容 | 统一帧头：Version + Type + Flags + StreamID + Length |
| **帧头最小长度** | 25字节 | 15字节 |
| **类型编码** | 1字节枚举 (0x01-0x05) | 62位变长整数 |
| **类型空间** | 256种 | 2^62种 |
| **VarInt实现** | 自定义编码，最大2^30-1 | 标准QUIC编码，最大2^62-1 |

### 消息类型对比

| V1 消息 | V2 消息 | 变化说明 |
|---------|---------|----------|
| `SQConnect` | `Connect` + `ConnectAck` | 新增ACK响应机制 |
| `SQBind` | (已移除) | 不再支持 |
| `SQAssociatOverDatagram` | `UdpAssociate` | 简化为单一类型 |
| `SQAssociatOverStream` | `UdpAssociate` | 统一为单一类型 |
| `SQAuthenticate` | `ClientHello` | 扩展为完整握手消息 |
| 无 | `ServerHello` | 新增服务端握手响应 |
| `SQUdpControlHeader` | `UdpData` | 简化为单一数据帧 |
| `SQPacketDatagramHeader` | `UdpData` | 合并到统一帧格式 |

### 握手流程对比

**V1 协议（有认证）：**
```
Client                                          Server
  |                                               |
  |--- [BIDI] SQAuthenticate ------------------->|
  |<-- [BIDI] (auth response) -------------------|
  |                                               |
  |--- [BIDI] SQConnect ------------------------>|
  |                                               |
  |=============== Data ==========================|
```
> 注：V1无认证时可直接发送SQConnect，1 RTT后开始数据传输

**V2 协议：**
```
Client                                          Server
  |                                               |
  |--- [BIDI] ClientHello ----------------------->|
  |<-- [BIDI] ServerHello -----------------------|
  |                                               |
  |--- [BIDI] Connect --------------------------->|
  |<-- [BIDI] ConnectAck ------------------------|
  |                                               |
  |=============== Data ==========================|
```
> 注：V2固定需要2 RTT才能开始数据传输（ClientHello+ServerHello + ConnectAck）

### TCP代理流程对比

**V1 协议：**
1. 客户端直接发送 `SQConnect` 请求
2. 服务端接收后直接建立TCP转发
3. 双向数据透传

**V2 协议：**
1. 客户端发送 `Connect` 请求
2. 服务端返回 `ConnectAck` 确认
3. 双向数据透传
4. 新增ACK机制确保请求被正确接收

### UDP传输对比

**V1 协议：**
- 控制消息：`SQUdpControlHeader` (目标地址 + ID)
- 数据消息：`SQPacketDatagramHeader` (ID) + 长度前缀数据
- 两种UDP模式：数据报模式(QUIC datagram) 和 流模式(QUIC unidirectional stream)

**V2 协议：**
- 统一使用 `UdpData` 帧
- 内置目标地址（首次携带，后续省略）
- 内置负载长度（VarInt编码）
- 简化实现，统一使用QUIC流

### V2 协议优势

#### 1. 编码效率提升

| 指标 | V1 | V2 | 提升 |
|------|-----|-----|------|
| 最小帧头 | 25字节 | 15字节 | -40% |
| 常见地址编码 | 7字节 | 7字节 | 持平 |
| VarInt范围 | 2^30-1 | 2^62-1 | 2^32倍 |

#### 2. 握手流程改进

| 指标 | V1 | V2 | 说明 |
|------|-----|-----|------|
| RTT数（无认证） | 1 RTT | 2 RTT | V2需要额外1 RTT |
| RTT数（有认证） | 2 RTT | 2 RTT | 持平 |
| 认证方式 | 单独消息 | 捎带在ClientHello | 减少消息数 |
| 可靠性 | 无确认 | ACK确认机制 | 更可靠 |

#### 3. 可扩展性增强

| 特性 | V1 | V2 |
|------|-----|-----|
| 类型空间 | 256种 | 2^62种 |
| 扩展机制 | 无 | TLV格式扩展字段 |
| 特性协商 | 无 | Hello消息特性位 |
| 多路复用 | 依赖QUIC | 内置32位流ID |

#### 4. 协议简化

| 方面 | V1 | V2 |
|------|-----|-----|
| UDP模式 | 2种（数据报/流） | 1种（统一流） |
| 消息格式 | 多种独立结构 | 统一Frame枚举 |
| 状态管理 | 复杂 | 简化 |

#### 5. 兼容性

| 特性 | V1 | V2 |
|------|-----|-----|
| VarInt标准 | 自定义 | IETF QUIC标准 |
| 地址格式 | SOCKS5 | SOCKS5（兼容） |
| 错误处理 | 不统一 | 统一错误码 |

|----------|-----|-----|------|
| 小请求延迟 | 50ms | 100ms | V2多1 RTT确认 |

### 迁移指南

V1和V2协议**不兼容**，需要同时升级客户端和服务端。

#### 客户端升级
1. 使用 `ClientHello` 替代 `SQAuthenticate`
2. 发送 `Connect` 后等待 `ConnectAck`
3. 使用 `UdpData` 帧发送UDP数据

#### 服务端升级
1. 解析 `ClientHello` 进行认证
2. 返回 `ServerHello` 进行特性协商
3. 收到 `Connect` 后返回 `ConnectAck`
4. 使用 `UdpData` 帧处理UDP数据
